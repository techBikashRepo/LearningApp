## SECTION 9 — Certification Focus (AWS SAA-C03)

```
CHOOSE EC2 WHEN:
  ✓ Stateful application that requires specific OS configuration
  ✓ Application needs GPU (ML inference, video processing)
  ✓ Memory > 30GB sustained (Lambda max = 10GB)
  ✓ Application runtime > 15 minutes in a single task (Lambda = 15 min max)
  ✓ Existing on-premises app needs lift-and-shift (Windows app, Oracle DB agent)
  ✓ Spot instances for batch workloads (ML training, HPC, rendering)
  ✓ You need to control OS kernel parameters, huge pages, NUMA topology

CHOOSE ECS FARGATE WHEN:
  ✓ Containerized application (Dockerfile exists or easy to create)
  ✓ Variable traffic with predictable container size (CPU/memory known)
  ✓ Don't want to manage OS patching, AMI updates, ASG configuration
  ✓ Microservices (multiple services → each runs as Fargate tasks)
  ✓ < 30GB memory needed per task

CHOOSE LAMBDA WHEN:
  ✓ Event-driven, short-lived tasks (S3 trigger, API Gateway, DynamoDB stream)
  ✓ Execution < 15 minutes
  ✓ Infrequent invocations (not worth reserving capacity 24/7)
  ✓ Zero infrastructure management desired
  ✓ Serverless API with API Gateway + Lambda (small teams, MVPs)

DECISION TREE:
  Is it containerized or containerizable?
    No → EC2 (lift-and-shift, OS-specific, GPU, huge memory)
    Yes →
      Is execution < 15 min and event-driven?
        Yes → Lambda
        No → ECS Fargate
      Do you need GPU?
        Yes → EC2 with GPU instance type (g4dn, p3, inf2)
        No → Fargate
```

## SECTION 10 — Comparison Table

```
INSTANCE PURCHASING STRATEGY:

TIER 1 — Baseline Production Load (always on, predictable):
  Choice: 1-year Compute Savings Plan (no instance type commitment)
  Savings: ~40% vs On-Demand
  Flexibility: covers EC2 + Lambda + Fargate (auto-applies across types)
  When: production app consistently running 24/7

TIER 2 — Steady-state w/ some flexibility needed:
  Choice: 1-year Standard Reserved Instance (specific AZ or Region)
  Savings: ~40-60% vs On-Demand
  When: you know exact instance type (e.g., always r6g.xlarge for DB)

TIER 3 — Dev/Test/Staging:
  Choice: On-Demand (spin up/down freely)
  Optimization: Instance Scheduler (automate start/stop 9am-6pm workdays)
  Savings: ~70% for non-24/7 environments

TIER 4 — Batch, CI/CD, fault-tolerant jobs:
  Choice: Spot Instances
  Savings: 70-90% vs On-Demand
  Conditions: job must handle 2-minute interruption notice + checkpoint state to S3
  Tools: Spot Fleet, AWS Batch (auto-handles Spot interruptions)

TIER 5 — Compliance/license requirements (dedicated hardware):
  Choice: Dedicated Host (fixed, per-host pricing)
  When: Windows/RHEL/Oracle BYOL licenses with per-socket/per-core terms

COST OPTIMIZATION CHECKLIST:
  Rightsizing:     AWS Compute Optimizer → EC2 recommendations (over/under provisioned)
  Idle detection:  CPU < 5% for 14 days → likely can terminate/rightsize
  EBS orphans:     delete unattached volumes, old snapshots (cost sink)
  Savings Plans:   cover baseline with SP, burst with On-Demand
```

## SECTION 11 — Quick Revision

```
TRAP 1: EBS volumes are AZ-locked
  EBS volume created in us-east-1a → can ONLY attach to EC2 in us-east-1a
  Cannot cross AZ. Cannot cross Region.
  To move: create snapshot → restore in new AZ
  Exam question: "EC2 in 1a needs data from EBS volume in 1b" → snapshot + restore

TRAP 2: Instance Store = lost on STOP (not just terminate)
  Reboot: data safe (same physical host)
  Stop + Start: data GONE (may land on different host)
  Exam: "which storage survives stop/start?" → EBS. Not instance store.

TRAP 3: t-series CPU Credits
  Spot high network traffic + t-series CPU throttle = common exam scenario
  Fix options: (a) enable t3 unlimited, (b) switch to m5 (fixed-performance)
  Exam: "suddenly slow, CPU at 80%" → check if t-series + CPUCreditBalance = 0

TRAP 4: Spot Instance = 2-minute warning, then terminated
  2-minute interruption notice via instance metadata + CloudWatch event
  Application must: checkpoint state to S3/EBS before 2 min expire
  Exam: "batch job must not lose progress" → use Spot + checkpoint to S3
  NOT Lambda (15 min limit) for multi-hour jobs

TRAP 5: Reserved Instance Scope — AZ vs Regional
  AZ-scoped RI: capacity reservation guaranteed in that AZ (important for launch)
  Regional-scoped RI: discount applies anywhere in region, NO capacity reservation
  Exam: "need guaranteed capacity during disaster recovery" → AZ-scoped RI

TRAP 6: AMI is Region-specific
  AMI created in us-east-1 cannot be directly used in eu-west-1
  Must: copy AMI to target region
  Exam: "multi-region deployment from same AMI" → copy AMI first

TRAP 7: EC2 Auto Scaling ≠ always running target capacity
  Desired=4, Min=2, Max=10
  If AZ goes down and reduces to 2 → AZ rebalancing will try to restore
  Scale-in termination: uses Default Termination Policy (oldest launch config first)
```

## SECTION 12 — Architect Thinking Exercise

```
COMPANY: E-commerce platform on physical servers (3× Dell PowerEdge, 256GB RAM, 32 cores)
REQUIREMENT: Migrate to AWS. Minimize downtime. Reduce cost. Prepare for future containers.
TRAFFIC: 50K req/day normal, 500K req/day on sale events (10×)

RECOMMENDED MIGRATION PATH:
  Phase 1 — Lift and Shift to EC2 (Month 1-2)
    │ Tool: AWS Application Migration Service (MGN)
    │ Replicate physical server to EC2 while live
    │ Cutover window: < 1 hour (minutes of final delta replication)
    │ Instance choice: r6i.8xlarge (256GB RAM, 32 vCPU = closest match)
    │ Purchasing: On-Demand during migration validation (1 month)
    │ Then: 1-year Compute Savings Plan for 40% discount
    │
  Phase 2 — Optimize and Instrument (Month 3)
    │ Add CloudWatch monitoring (CPUUtilization, MemoryUsed via agent)
    │ Move database off-EC2 → RDS PostgreSQL (separate scaling)
    │ Static assets → S3 + CloudFront (remove I/O from EC2)
    │ Session state → ElastiCache Redis (remove stickiness requirement)
    │
  Phase 3 — Auto Scaling (Month 4)
    │ Stateless app now (session in Redis) → can scale horizontally
    │ Create AMI from EC2 (golden AMI) via AMI pipeline
    │ Launch Template: r6i.2xlarge with gp3 EBS
    │ ASG: Min=2, Desired=3, Max=15, scale on CPU > 70%
    │ Sale events: scheduled scaling (8AM→Max=20 for Black Friday)
    │
  Phase 4 — Containerize (Month 6-12)
    Identify stateless microservices → Dockerize
    Run on ECS Fargate (no EC2 management)
    Core monolith: stays on EC2 until fully decomposed

COST COMPARISON:
  Physical servers: $15,000/month (amortized hardware + datacenter)
  Phase 1 EC2: $8,400/month (r6i.8xlarge On-Demand × 2)
  Phase 3 ASG: $4,200/month average (Savings Plans + rightsize to 2xlarge)
  Phase 4 ECS: $2,800/month average (Fargate pay-per-task)
```

### Architect's Mental Model

```
5 RULES I NEVER VIOLATE:

1. "t-series is a dev/test series — production APIs always get m-series or better"
   t3 has ceiling. m5 has none. In a crisis, the ceiling hits you first.

2. "EBS volumes are tenants of their AZ — plan cross-AZ data movement explicitly"
   Nothing frustrates more than architecting a failover without accounting for EBS.
   Multi-AZ: use EFS (shared NFS) or RDS (managed replication). Not raw EBS.

3. "Instance Store = working memory, EBS = disk, S3 = long-term memory"
   Never rely on instance store for anything that matters.
   Default: EBS. Instance store only if raw I/O benchmark requires it, with S3 checkpoint.

4. "A Launch Template is the single source of truth for the instance fleet"
   No snowflake EC2 instances. Everything reproducible from LT + golden AMI.
   If you can't terminate any instance and rebuild it in 5 minutes, it's a risk.

5. "Use SSM Session Manager — no SSH, no bastion, no port 22 open"
   Every open port is an attack surface. SSM over HTTPS port 443 (already open).
   Eliminates bastion cost, key management overhead, and audit gaps.

3 MISTAKES JUNIOR ARCHITECTS MAKE:

1. Choosing EC2 for everything
   "We know EC2 so we'll use EC2"
   Result: managing OS patches, AMI pipelines, ASG configs that ECS handles for free.
   Question first: is this containerizable? Then default to ECS.

2. No rightsizing cadence
   Provision r6i.4xlarge "to be safe." CPU avg = 8%. Cost × 4 vs what's needed.
   AWS Compute Optimizer: check weekly. Rightsizing saves 40% on typical fleets.

3. Forgetting the EBS AZ boundary during disaster recovery design
   "We'll failover to us-east-1b if 1a goes down."
   EBS volumes in 1a: can't be attached in 1b. Snapshots take 20 minutes.
   Fix: RDS for DB (Multi-AZ managed), EFS for shared files, or pre-staged snapshots.

30-SECOND MENTAL MODEL (Say this in an interview):
  "EC2 is a virtual machine. You manage the OS, we manage the hardware.
   Choose EC2 when you need OS control, GPU, or large memory. Otherwise:
   containers → ECS Fargate, event-driven → Lambda.
   For EC2 fleets: Launch Templates + AMIs + Auto Scaling Groups.
   Purchasing: Savings Plans for baseline, Spot for fault-tolerant batch.
   Core rule: every EC2 instance must be replaceable — immutable infrastructure."
```
