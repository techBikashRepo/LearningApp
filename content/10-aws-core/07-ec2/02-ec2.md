## SECTION 5 — Real World Example

```
INCIDENT TIMELINE:
T+0:   Black Friday promotion starts. Traffic 3× normal.
T+30:  API response time jumps from 80ms to 2,400ms. Timeout errors begin.
T+45:  Auto Scaling Group adds t3.large instances (correct scaling response).
T+50:  New instances don't help. Response time still 2,400ms.
T+60:  CPU metrics show: 100% CPU utilization on ALL instances. Including new ones.
T+70:  Root cause: t3 instances exhausted CPU credits 30 minutes into the traffic spike.

ROOT CAUSE:
  t3 (burstable) instances:
    Baseline CPU: t3.large = 30% of 2 vCPUs = 0.6 vCPUs continuously
    Above 30%: uses accumulated CPU credits
    Credits: earn 36 credits/hour when below baseline
    Burst: spend credits at 1 credit per vCPU-minute above baseline

  When credits exhausted:
    If T3 in standard mode: performance throttled TO baseline (30% CPU)
    4×-traffic → application needs 100% CPU → throttled to 30% → 3× slower
    Adding more throttled instances doesn't help proportionally

  t3 unlimited mode:
    Can burst indefinitely, charged $0.05/vCPU-hour for surplus credits
    Better for production: never throttled, slight extra cost during bursts
    Or: for predictable high load → switch to m-series (no credit concept)

FIX:
  Immediate: enable T3 Unlimited on Auto Scaling Group
    aws ec2 modify-instance-credit-specification \
      --instance-credit-specification InstanceId=i-xxxx,CpuCredits=unlimited

  Permanent: switch to m6g.large (same price range, consistent 100% CPU available)

DETECTION:
  CloudWatch metric: CPUCreditBalance → alert when < 20 credits (1 vCPU-minute = 1 credit)
  CPUSurplusCreditsCharged > 0 → burning through credits, about to throttle

PREVENTION:
  Production API servers: default to m-series, not t-series
  t-series: only for dev/test environments with predictable low workload
```

## SECTION 6 — System Design Importance

```
INCIDENT:
  Production application loses in-flight transactions during scale-in event.
  Auto Scaling Group scales down from 6 to 4 instances.
  2 instances terminated while processing requests → 503 for those requests.

ROOT CAUSE:
  Auto Scaling default termination policy: terminates "oldest instance first"
  ALB draining: default connection draining deregistration delay = 300 seconds
  But: Auto Scaling issued terminate WITHOUT waiting for draining to complete

  Sequence:
  T+0: Scale-in event triggered
  T+1: Auto Scaling selects 2 instances to terminate
  T+2: Deregistration initiated (ALB starts draining)
  T+3: Auto Scaling terminates instances IMMEDIATELY (didn't wait for drain)
  T+3: In-flight requests to those instances → connection reset → 503

CORRECT CONFIGURATION:
  1. Auto Scaling + Instance Warmup: configure lifecycle hooks

  aws autoscaling put-lifecycle-hook \
    --auto-scaling-group-name my-asg \
    --lifecycle-hook-name instance-terminating \
    --lifecycle-transition autoscaling:EC2_INSTANCE_TERMINATING \
    --heartbeat-timeout 300 \
    --default-result CONTINUE

  Lifecycle hook: before terminating → instance goes to "Terminating:Wait" state
  Lambda/SNS triggers: can signal CONTINUE when draining complete

  2. ALB deregistration delay: ensure = 300 seconds (default, but confirm)

  3. Auto Scaling termination + ALB draining integration:
     Modern approach: set instance scale-in protection during active request processing
     ECS Fargate: handles this automatically (task termination grace period)

BETTER ARCHITECTURE: Use ECS instead of directly managing EC2 in ASG
  ECS task termination: SIGTERM → graceful shutdown period → SIGKILL
  ECS handles ALB deregistration + task drain automatically
  Lifecycle management is built-in, not custom
```

## SECTION 7 — AWS & Cloud Mapping

```
INCIDENT:
  ML team stores training artifacts on i3.large instance store (NVMe).
  Team stops instance on Friday evening (to save cost over weekend).
  Monday: instance restarted. All training artifacts gone.

ROOT CAUSE:
  Instance store is physically attached to the EC2 host.
  "Stop" instance: instance moves off the host (goes to stopped state).
  When started: may launch on a DIFFERENT physical host.
  Previous host's NVMe: no longer accessible. Data permanently lost.

  Instance store behavior:
    ├── EC2 reboot (sudo reboot): data PRESERVED (same host, same NVMe)
    ├── EC2 stop/start: data LOST (host may change)
    ├── EC2 termination: data LOST
    └── Hardware failure (host goes down): data LOST

FIX:
  Anything that must persist: EBS volume only
  Instance store: strictly ephemeral scratch space

  For ML training artifacts:
    Write to EBS (persists through stop/start) or S3 (durable, cheap)
    S3: aws s3 sync ./artifacts s3://my-bucket/artifacts/ during training

CHANGED ARCHITECTURE:
  Before: i3.large with local NVMe → fast, but data lost on stop
  After: r6id.large (memory + local NVMe for temp) + EBS for persistent + S3 checkpoint
  Cost: $0.02/GB EBS vs $0 for instance store
  Trade: $2/month per 100GB vs data loss risk on every stop
  Obvious choice: EBS for persistence

GOLDEN RULE: Instance store = temporary. EBS = persistent. S3 = archival.
```

## SECTION 8 — Interview Preparation

**Beginner:**

**Q: What is EC2 and when would you choose it over ECS/Fargate?**
**A:** EC2 (Elastic Compute Cloud) provides virtual servers in the cloud. You're renting a slice of hardware â€” you choose the OS, install software, and manage it. Choose EC2 when: (1) You need full OS control (custom kernel parameters, specific software not available in containers). (2) Running workloads that need long-lived state on disk (CI/CD build agents with large caches). (3) Running non-containerized legacy apps. Choose ECS/Fargate over EC2 when: you're deploying modern containerized apps â€” Fargate manages the underlying EC2 instances for you, elastic scaling is simpler, no OS patching burden. Most new backend services should default to ECS Fargate.

**Q: What is an EC2 instance type and how do you choose the right one?**
**A:** Instance type defines the CPU, memory, storage, and networking capacity. Format: {family}{generation}.{size}. Examples: m7g.large = general purpose, 7th gen, graviton (ARM), large size (2 vCPU, 8GB RAM). Families: *m* = general purpose (balanced CPU/memory), *c* = compute optimized (high CPU, less memory â€” good for computation), *r* = memory optimized (more RAM â€” good for caches, large in-memory datasets), *t* = burstable (cheap, for low-traffic workloads with occasional spikes). For a typical Node.js API: start with 	3.small or m7g.medium. Right-size using CloudWatch CPU and memory metrics after running for 1 week.

**Q: What is an EC2 AMI and what is its role in deployments?**
**A:** An AMI (Amazon Machine Image) is the template for launching an EC2 instance â€” it contains the OS, pre-installed software, and initial configuration. When you launch an EC2, you pick an AMI: Amazon Linux 2023, Ubuntu 22.04, or a custom AMI your team built. For modern microservices on Docker/ECS: you typically just use the official Amazon Linux/Ubuntu AMI and let Docker handle application deployment (not baked into the AMI). AMI-based deployment (Packer Baking): bake your application into a custom AMI, launch new AMIs for each deployment. Used for: performance-sensitive apps (no Docker overhead), teams using HashiCorp Packer/Terraform for immutable server deployments.

**Intermediate:**

**Q: What are the EC2 pricing models and when should you use each?**
**A:** *On-Demand:* pay by the hour/second, no commitment. Flexible, expensive long-term. Use for: unpredictable workloads, development/staging. *Reserved Instances:* 1 or 3 year commitment, 40-75% discount. Use for: predictable baseline load (e.g., "we always need 5 EC2 instances"). *Spot Instances:* bid on spare AWS capacity, up to 90% discount. Can be interrupted with 2-minute warning. USE for: fault-tolerant batch processing, stateless workers, CI/CD builds. NEVER for: databases, anything that can't tolerate interruption. *Savings Plans:* commit to a spend amount per hour for 1-3 years (more flexible than Reserved Instances â€” committed to compute spend, not specific instance type).

**Q: What is EC2 instance store vs EBS and when does it matter?**
**A:** *EBS (Elastic Block Store):* persistent block storage attached to EC2. Survives instance stop/start. Can be detached and reattached. Backed up via snapshots. Use for: OS volume, databases on EC2, any data that must survive instance restart. *Instance Store:* physical disks directly attached to host hardware. Ephemeral â€” data is LOST when instance stops, terminates, or the host hardware fails. Much faster (NVMe SSD directly attached). Use for: temp files, caches, scratch space. Common mistake: running a database on instance store and losing all data after instance maintenance/stop. Rule: never store important data on instance store.

**Q: What is the EC2 metadata service and what is the SSRF risk?**
**A:** The EC2 metadata service (http://169.254.169.254/latest/meta-data/) provides instance metadata and temporary IAM credentials to processes running on the instance. Any process that can make HTTP requests can access it â€” including your web application. If your app has an SSRF (Server-Side Request Forgery) vulnerability â€” an attacker tricks your app into making an HTTP request to 169.254.169.254/latest/meta-data/iam/security-credentials/{role-name} and reads the AWS temporary credentials. Mitigation: IMDSv2 (Metadata Service Version 2) â€” requires a session token that needs a PUT request first, making SSRF more difficult. Enable IMDSv2 only: ws ec2 modify-instance-metadata-options --http-tokens required.

**Advanced (System Design):**

**Scenario 1:** Design an autoscaling strategy for a web application with predictable daily traffic patterns (low traffic 2-6 AM, peak 12-2 PM and 6-9 PM). The app uses ECS on EC2 launch type (not Fargate) due to cost requirements. Design the autoscaling configuration.

*EC2 Auto Scaling Group with EC2 launch type:*
(1) *Scheduled Scaling:* two scale-in/scale-out actions on schedules:
- 1:00 AM: scale to minimum capacity (2 instances for HA)
- 11:30 AM: pre-scale to 6 instances (15 min before lunch peak)
- 3:00 PM: scale back to 3 instances
- 5:30 PM: pre-scale to 8 instances (before evening peak)
- 10:00 PM: scale back to 2 instances

(2) *Dynamic Target Tracking Scaling* (on top of scheduled): CPU Target = 65% â€” if lunch peak is higher than expected, auto-scaling adds instances beyond the scheduled amount.

(3) *ECS Capacity Provider:* link EC2 ASG to ECS capacity provider â€” ECS task placement drives EC2 scaling (scales EC2 instances based on ECS task scheduling needs, not just CPU).

(4) *Warm pools:* pre-warm stopped instances â€” reduces launch time from 3 min to 30s during peak.

**Scenario 2:** Your EC2-based ECS cluster is showing 80% CPU at peak. Some engineers suggest migrating to Fargate, others want to add more EC2 instances. Analyze the trade-offs and make a recommendation.

*EC2 clusters vs Fargate trade-off:*
*EC2 current situation:*
(+) Existing Reserved Instance discounts (if any) â€” sunk cost already paid.
(-) 80% CPU = near capacity, hitting ceiling, adds operational risk.
(-) You manage EC2 patching, AMI updates, capacity planning.

*Adding EC2 instances:*
(+) Cheaper if using Reserved Instances.
(-) Capacity planning complexity grows. Bin-packing efficiency problem.
(-) Still your responsibility to maintain.

*Migrating to Fargate:*
(+) No capacity planning â€” Fargate is elastic.
(+) No EC2 patching/management.
(+) Per-task CPU/memory billing â€” you pay exactly for what tasks need.
(-) ~20-30% more expensive per vCPU-hour than Reserved EC2.
(-) Cold start latency for scaling events (Fargate task launch ~30s vs EC2 pre-warmed container: ~5s).

*Recommendation:* If cluster has < 50 tasks and team is small: migrate to Fargate (eliminate operational burden). If running > 100 tasks with consistent load and Reserved Instances in place: keep EC2 with proper capacity management, fix the 80% saturation by right-sizing (switch to larger instance types with better bin-packing, use Spot Instances for ~30% of tasks).
