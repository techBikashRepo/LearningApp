## SECTION 1 — Intuition (Explain Like a 12-Year-Old)

```
PHYSICAL EQUIVALENT:
  EC2 instance → Physical server (Dell/HP/Supermicro)
  EC2 instance type → Server specification (CPU, RAM, NIC speed, storage)
  EC2 AMI → OS disk image backup / server template
  EBS volume → SAN/NAS disk (network-attached block storage)
  Instance store → Local NVMe disk (physically on the host)
  Auto Scaling Group → Server cluster with load balancer and capacity management

  Physical:
    Server procurement: 6-12 week lead time
    Capacity planning: buy for peak + 1 year ahead
    Hardware failure: swap physical hardware, manual intervention
    Scaling: rack, cable, configure new servers
    Cost: CAPEX ($5,000–$50,000+ per server) + data center colocation

  EC2:
    Provisioning: API call → new instance in < 60 seconds
    Capacity: spawn/terminate as needed (pay per second/hour)
    Hardware failure: AWS handles → instance moves to healthy host
    Scaling: Auto Scaling Group increases/decreases count automatically
    Cost: OPEX (pay for what you use, RI/Savings Plans for committed discount)

MIN VIABLE MIGRATION FROM PHYSICAL TO EC2:
  Physical server running Ubuntu + Node.js app → AMI (image current server) → EC2 launch
  This is the simplest lift-and-shift. Not optimal, but it works.
  Next step optimization: containerize (ECS/EKS) for better density and portability
```

## SECTION 2 — Core Technical Explanation

```
INSTANCE TYPE FORMAT:  [family][generation][modifiers].[size]
  c7g.xlarge → c=compute optimized, 7=generation, g=Graviton (ARM), xlarge=size

FAMILY GUIDE:

GENERAL PURPOSE (balanced CPU/memory/network):
  t-series (t3, t4g): burstable, low baseline CPU, earns/spends CPU credits
    t3.micro: 2 vCPU, 1 GB RAM — FREE TIER eligible
    t3a.medium: AMD variant (cheaper ~10%)
    Best for: dev/test, low-traffic web, variable workloads
    TRAP: CPU credit exhaustion → performance throttle in production (use unlimited mode or switch to m-series)

  m-series (m6i, m6g): consistent CPU, no throttle
    m6i.large: 2 vCPU, 8 GB RAM — ~$70/month
    m6g: Graviton (ARM) — ~20% cheaper, same performance
    Best for: web servers, application servers, small DBs

COMPUTE OPTIMIZED (high CPU-to-memory ratio):
  c-series (c6i, c7g):
    c6i.large: 2 vCPU, 4 GB RAM — high-freq Intel, ENA networking
    Best for: game servers, batch processing, video encoding, CPU-intensive APIs
    NOT for: databases (need more memory per CPU)

MEMORY OPTIMIZED (high memory-to-CPU ratio):
  r-series (r6i, r7g): 8 GB RAM per vCPU
    r6i.large: 2 vCPU, 16 GB RAM — ~$95/month
    Best for: in-memory databases, Redis (self-hosted), large JVM heaps

  x-series: extreme memory (up to 24 TB RAM) — SAP HANA, large analytics

STORAGE OPTIMIZED (high I/O, NVMe local storage):
  i-series: i3.large has NVMe SSDs — for databases needing < 100 μs latency
  d-series: dense HDD storage — Hadoop, data warehouses

ACCELERATED COMPUTING:
  g-series (g4dn, g5): NVIDIA GPU — ML inference, video transcoding
  p-series (p3, p4): NVIDIA high-end GPU — ML training, HPC
  inf-series: AWS Inferentia chip — low-cost ML inference
  trn-series: AWS Trainium — ML training (cheaper than p-series)

GRAVITON (ARM) INSTANCES:
  Suffix "g": t4g, m6g, c6g, r6g
  AWS-designed ARM-based CPU
  25-40% better price-performance vs x86 equivalent
  Supported by: Linux, Docker, most languages (Python, Node.js, Java, Go natively)
  NOT supported: Windows binaries, x86-only compiled software
  Recommendation: default choice for new workloads on Linux

SIZE SUFFIXES:
  nano < micro < small < medium < large < xlarge < 2xlarge < 4xlarge < ... < metal
  Each step roughly doubles vCPU and memory
  metal: bare-metal (no hypervisor overhead, dedicated for compliance or performance)
```

## SECTION 3 — Architecture Diagram (MANDATORY)

```
ON-DEMAND:
  Pay per second (minimum 60 seconds) for Linux/Windows
  No commitment, full flexibility
  Most expensive per-hour rate
  Use for: variable workloads, testing, spiky traffic, short-lived jobs

RESERVED INSTANCES (RI):
  1-year or 3-year commitment
  Payment options: All Upfront (max discount), Partial Upfront, No Upfront
  Discount: up to 72% off On-Demand
  Standard RI: specific instance type in specific AZ or region
  Convertible RI: can exchange for different instance type/OS midterm

  1-year no-upfront savings:
    m6i.large On-Demand: ~$70/month
    m6i.large 1-year RI:  ~$46/month (34% savings)
    m6i.large 3-year RI:  ~$29/month (59% savings)

  Use for: steady-state workloads running 24/7 (database servers, always-on APIs)

SAVINGS PLANS:
  Commitment: $/hour spend not specific instance type
  Compute Savings Plan: most flexible — any instance family/size/region/OS
  EC2 Savings Plan: specific family in region, any size/OS
  Discount: up to 66% off On-Demand (slightly less than RI but more flexible)
  Use for: predictable but changing workloads (migrating to containers, changing instance types)

SPOT INSTANCES:
  Use AWS's spare capacity — price fluctuates with supply/demand
  Discount: up to 90% off On-Demand (significant savings)
  Risk: AWS can reclaim with 2-minute notice (Spot interruption)
  NOT for: stateful workloads, production databases, anything that can't restart
  Best for: batch jobs, data processing, ML training, stateless workers

  Spot pricing example:
    m6i.large On-Demand: $0.096/hour
    m6i.large Spot: ~$0.03/hour (68% savings)
    8 hours of batch job: $0.25 vs $0.77 (significant for large workloads)

DEDICATED HOSTS:
  Entire physical host dedicated to your account
  Compliance: visible server socket/core for licensing (Windows Server, SQL Server per-core)
  No other customer workloads on same host
  Cost: server-level pricing (much higher), but software license savings can justify

BILLING CALCULATION:
  On-Demand t3.medium (2 vCPU, 4GB) in ap-south-1: $0.0416/hour
  24 hours/day × 30 days = 720 hours × $0.0416 = $29.95/month
  With 1-year RI: $19.60/month
  With Savings Plan: ~$21/month
```

## SECTION 4 — Request Flow (MANDATORY FLOWCHART)

```
STORAGE TYPE         LATENCY   PERSISTENCE   USE CASE
──────────────────────────────────────────────────────────────────
EBS (gp3)            < 1ms     Persists independently of instance  OS disk, DB data
EBS (io2 Block Exp)  < 0.1ms   Persists independently               High-IOPS DB
Instance Store       < 0.1ms   LOST on stop/terminate               Temp processing
EFS (NFS)            1-10ms    Shared across instances              Shared file storage
FSx for Lustre       < 1ms     High-performance shared              ML, HPC
──────────────────────────────────────────────────────────────────

EBS VOLUME TYPES:
  gp3 (General Purpose SSD): default for most use cases
    3,000 IOPS baseline (up to 16,000 IOPS independently of size)
    125 MB/s throughput (up to 1,000 MB/s)
    Cost: $0.08/GB/month

  gp2 (legacy): IOPS tied to volume size (3 IOPS/GB — migrate to gp3, same cost, better perf)

  io2 Block Express: for I/O-intensive DBs (PostgreSQL, MySQL, Oracle)
    Up to 256,000 IOPS, sub-millisecond latency
    Cost: $0.125/GB/month + $0.065/provisioned IOPS-month

  st1 (HDD): throughput-optimized, large sequential reads
    Log processing, data warehouses, big data
    Cost: $0.045/GB/month (cheaper, lower IOPS)

  sc1 (cold HDD): archival, rarely accessed
    Cost: $0.015/GB/month

INSTANCE STORE:
  Physical NVMe drives on the EC2 host (not network-attached)
  Ultra-low latency (no network hop)
  No additional cost (included with instance type)
  CRITICAL: all data LOST on: instance stop, instance termination, hardware failure
  Use for: scratch space, temporary caches, buffer/queue processing
  NOT for: any data you need to keep
  EC2 reboot: data preserved. EC2 stop/start: data lost.
```

### EC2 In the Three-Tier Architecture

```
LAYER           SERVICE            SIZING GUIDANCE
──────────────────────────────────────────────────────────────────────
Presentation    ALB (managed)      No EC2 sizing — AWS managed
Application     EC2 (or ECS/EKS)   CPU-bound: c6i/c7g | Memory-bound: m/r series
Data            RDS (or EC2)       Prefer RDS > self-managed DB on EC2
                                   (if self-managed: r-series for memory)
──────────────────────────────────────────────────────────────────────

EC2 vs ECS/EKS vs Lambda for Application Tier:

EC2 (direct):
  ✅ Full OS control (custom kernel, packages, compliance)
  ✅ Predictable performance (no cold start, no container overhead)
  ✅ Long-running processes (WebSocket servers, daemon processes)
  ❌ You manage OS patches, AMI updates, bootstrap scripts
  ❌ Instance-level scaling (slower than container scheduling)
  Best for: legacy apps, Windows workloads, heavy databases (self-managed)

ECS Fargate (containers on managed EC2):
  ✅ No OS management (AWS manages host layer)
  ✅ Bin-packing: multiple tasks on one host (better utilization)
  ✅ Faster autoscaling (task scheduling faster than instance launch)
  ❌ Less control over OS tuning
  Best for: microservices, APIs, stateless workloads

Lambda:
  ✅ Zero infrastructure management
  ✅ True per-invocation billing (no idle cost)
  ❌ Cold start latency (100ms-5s for first invocation)
  ❌ Max 15-minute execution time, 10GB RAM
  Best for: event-driven processing, scheduled tasks, API with spiky traffic
```

### AMI and Launch Template

```
AMI (Amazon Machine Image):
  Template for EC2 instance: OS + pre-installed software + configuration
  Components:
    ├── Root EBS snapshot (OS disk)
    ├── Block device mapping (which EBS volumes to attach on launch)
    ├── Virtualization type (HVM — hardware virtual machine, for modern instances)
    └── Architecture (x86 or ARM/Graviton)

  Types:
    AWS-provided: Amazon Linux 2023, Ubuntu, Windows Server, RHEL
    AWS Marketplace: commercial software pre-installed (Nginx Plus, Datadog, etc.)
    Custom (golden AMI): your team builds → includes app, config, agents pre-installed

  Golden AMI pattern (production best practice):
    Base: AWS Amazon Linux 2023 → Install: CloudWatch agent, SSM agent, app dependencies
    Bake: everything in image (no bootstrapping at launch time)
    Fast start: instance ready in < 60 seconds (vs 2-5 min bootstrapping on start)
    Immutable: never patch running instances, rebuild AMI, roll new instances

LAUNCH TEMPLATE:
  Modern way to define EC2 launch configuration
  Contains: instance type, AMI ID, key pair, security groups, user data, IAM role, EBS config
  Versioned: can have multiple versions, specify $Default or $Latest
  Used by: Auto Scaling Groups, EC2 fleet, direct launch

  Best practice: use Launch Templates (not Launch Configurations — deprecated)
  Store in Terraform or CloudFormation for IaC
```

### Cost Model

```
EC2 COST COMPONENTS:
  1. Instance compute: $/hour (by instance type + region + OS)
  2. EBS storage: $/GB/month + $/IOPS provisioned
  3. Data transfer: free within AZ, $0.01/GB cross-AZ, $0.09/GB to internet
  4. Elastic IP when unassociated: $0.005/hour (~$3.65/month waste)
  5. EBS snapshots: $0.05/GB/month (only changed blocks stored)

EXAMPLE MONTHLY BILL (typical Node.js app server):
  m6g.large (Graviton): 2 vCPU, 8 GB RAM
    On-Demand: $0.077/hour × 720 = $55.44/month
    1-year RI No-upfront: $0.049/hour × 720 = $35.28/month
    3-year RI All-upfront: $0.030/hour × 720 = $21.60/month

  EBS gp3 root: 30 GB: $0.08 × 30 = $2.40/month

  Total with RI: $35.28 + $2.40 = $37.68/month per instance

COST GOVERNANCE:
  ├── Tag all EC2 with: Environment, Project, Owner, CostCenter
  ├── AWS Cost Explorer: group by tag to see per-team/project cost
  ├── EC2 rightsizing recommendations: CloudWatch CPU < 10% average → downsize
  ├── Delete unattached EBS volumes (a common waste source — ~$0.08/GB/month)
  └── Release unassociated Elastic IPs ($3.65/month/each × 50 = $182/month waste)
```
