## SECTION 1 — Intuition (Explain Like a 12-Year-Old)

```
PHYSICAL WORLD EQUIVALENT:
  Traditional: NAS/SAN array in data center. Expensive. Fixed capacity. Manual scaling.
  AWS S3: infinite-scale object storage. Pay per GB stored + requests.

WITHOUT S3:
  EC2 stores user uploads on EBS.
  EBS: AZ-locked. EC2 fails → 1 instance has the files. Cannot share across instances.
  Scale: 5 EC2 instances → files on instance 1, instances 2-5 don't have them.

WITH S3:
  Upload: EC2-1 writes file → S3 bucket
  Download: EC2-1, EC2-2, EC2-5, Lambda, anyone with permissions → reads same S3 object
  Decoupled: storage independent of compute. Scale compute freely.

KEY PROPERTIES:
  ├── Object storage (not block, not file system)
  ├── 11 nines durability (99.999999999% — AWS stores data across minimum 3 AZs)
  ├── 99.99% availability (Standard tier)
  ├── Virtually unlimited capacity (no pre-provisioning)
  ├── Global service — bucket in a region, accessible globally
  └── HTTP/HTTPS API (not mount, not attach — PUT/GET/DELETE operations)
```

## SECTION 2 — Core Technical Explanation

```
OBJECT MODEL:
  Bucket: top-level container. Name globally unique across all AWS accounts.
  Object: a file. Max size 5TB. Min size 0 bytes.
  Key: the full path/name of the object. e.g.: "images/user/avatar/123.jpg"
  No real "folders": flat namespace. "/" in key = simulated folder (UI shows as folder)

  Full object address: s3://my-bucket/images/user/avatar/123.jpg
  HTTPS URL: https://my-bucket.s3.amazonaws.com/images/user/avatar/123.jpg

BUCKET NAMING RULES:
  ├── Globally unique (across all AWS accounts, all regions)
  ├── 3-63 characters, lowercase, numbers, hyphens
  ├── Must start with letter or number
  └── No dots (breaks SSL cert validation)

STORAGE CLASSES:
  Class               | Use Case                    | Retrieval    | Cost/GB-month
  --------------------|-----------------------------|--------------|--------------
  S3 Standard         | Frequently accessed data    | Milliseconds | $0.023
  S3 Standard-IA      | Infrequent access (1/month) | Milliseconds | $0.0125 + retrieval fee
  S3 One Zone-IA      | Reproducible infrequent     | Milliseconds | $0.01 (1 AZ only)
  S3 Glacier Instant  | Archive, access 1/quarter   | Milliseconds | $0.004
  S3 Glacier Flexible | Archive, access planned     | Min-12 hours | $0.0036
  S3 Glacier Deep     | Rare access, compliance     | 12-48 hours  | $0.00099
  S3 Intelligent-Tier | Unknown/variable access     | Milliseconds | Auto-optimized

VERSIONING:
  Default: OFF. New upload with same key → overwrites previous.
  Versioning ON: every upload creates new version. Delete = delete marker (recoverable).
  Cannot disable versioning once enabled (can only suspend)
  MFA Delete: require MFA to permanently delete a versioned object
```

## SECTION 3 — Architecture Diagram (MANDATORY)

```
ACCESS CONTROL LAYERS (evaluated in order):

  Layer 1: Bucket Public Access Block (account-level OR bucket-level)
    Default: ALL public access blocked. This overrides everything below.
    Must explicitly disable "Block Public Access" to allow any public access.

  Layer 2: Bucket Policy (resource-based policy on the bucket)
    JSON policy. Controls access from IAM users, roles, other accounts, services.
    Attach to bucket. Evaluated for every request.

  Layer 3: IAM Policy (identity-based policy on the user/role)
    Who can do what. Evaluated with bucket policy (AND logic for same account).

  Layer 4: ACLs (legacy, not recommended)
    Object-level or bucket-level. Avoid for new architectures.

COMMON BUCKET POLICY — Allow public read (for static website):
  {
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::my-bucket/*"
    }]
  }

COMMON BUCKET POLICY — Allow only specific IAM role:
  {
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": {"AWS": "arn:aws:iam::123456789012:role/AppRole"},
      "Action": ["s3:GetObject", "s3:PutObject"],
      "Resource": "arn:aws:s3:::my-bucket/*"
    }]
  }

S3 ENCRYPTION:
  SSE-S3: AWS manages keys (AES-256). Default. Free.
  SSE-KMS: Customer-managed key via KMS. Audit key usage. ~$0.03/10K requests.
  SSE-C: Customer provides key on each request. AWS never stores the key.
  Client-side: encrypt before upload. Fully customer-managed.

  Default encryption: set at bucket level → all new objects encrypted automatically
```

## SECTION 4 — Request Flow (MANDATORY FLOWCHART)

```
PRODUCTION ARCHITECTURE: Three-Tier App + S3

  User → CloudFront CDN → [Cache Hit: return cached object]
                       ↓ [Cache Miss]
                      S3 Bucket (origin)

  Benefits:
    CDN (CloudFront): global edge caching → lower latency + lower S3 GET costs
    S3 origin: durable storage. CloudFront serves most reads.

  Cost example:
    1M downloads/day of 1MB files = 1TB/day
    S3 direct: 1TB × $0.09/GB = $92/day
    CloudFront cache hit 90%: 100GB from S3 × $0.09 + 900GB CloudFront = $9.2 + $8.1 = $17.3/day
    Savings: ~81%

S3 LIFECYCLE POLICIES (automated cost optimization):

  Rule 1: transition recent uploads through storage tiers
    Days 0-30:  S3 Standard          ($0.023/GB)
    Days 31-90: S3 Standard-IA       ($0.0125/GB)
    Days 91+:   S3 Glacier Instant   ($0.004/GB)
    Days 365+:  S3 Glacier Deep      ($0.00099/GB)

  Rule 2: delete old versions after N days (with versioning enabled)
  Rule 3: delete incomplete multipart uploads (> 7 days old) → common cost leak

S3 CROSS-REGION REPLICATION (CRR):
  Automatic async replication to bucket in another region
  Use cases: disaster recovery, serve users from nearby region, compliance
  Requires: versioning enabled on both source and destination
  Cost: replication data transfer ($0.02/GB cross-region)
```

### Cost Model

```
COMPONENTS:
  Storage:       charged per GB per month by storage class
  Requests:      PUT/COPY/POST: $0.005/1K | GET/SELECT: $0.0004/1K
  Data transfer: into S3: FREE | out to internet: $0.09/GB (first 10TB)
                 S3 to EC2 in SAME region: FREE
                 S3 to CloudFront: FREE (key optimization)

COST OPTIMIZATION STRATEGY:
  1. CloudFront in front of S3 → reduce S3 GET requests + eliminate egress charge
  2. Lifecycle policies → auto-tier to cheaper storage classes
  3. S3 Intelligent-Tiering → auto-optimize if access patterns unknown ($0.0025/1K monitoring)
  4. Delete incomplete multipart uploads (can be several GB of orphaned parts)
  5. S3 Storage Lens → visibility into usage across all buckets

COST TRAP:
  Versioning enabled + no lifecycle policy for old versions
  → every object version stored indefinitely
  → 1 year × 100K writes/day to 1MB files = 36.5TB of old versions
  → $840/month for versions alone
  Fix: lifecycle rule → delete non-current versions after 90 days
```
