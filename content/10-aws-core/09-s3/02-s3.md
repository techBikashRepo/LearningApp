## SECTION 5 — Real World Example

```
INCIDENT:
  Security audit flags S3 bucket "user-avatars-prod" as publicly accessible.
  Bucket contains 2.3M user profile photos — all publicly readable without authentication.

ROOT CAUSE:
  Developer needed to serve user photos publicly (via direct S3 URL).
  Action: disabled "Block Public Access" on bucket + added public GetObject bucket policy.
  Problem: used bucket policy with "Principal: *" and no condition.

  All user data exposed:
    https://user-avatars-prod.s3.amazonaws.com/users/12345/avatar.jpg → accessible by anyone
    Content included user IDs in object keys → enumerable

CORRECT ARCHITECTURE:
  ❌ Wrong: Public S3 bucket with direct URL access

  ✓ Correct Option A: CloudFront + OAC (Origin Access Control)
    CloudFront distribution → S3 origin (private bucket)
    S3 bucket policy: allow only CloudFront service principal
    User gets: https://cdn.example.com/users/12345/avatar.jpg (via CloudFront)
    S3 bucket: fully private, not directly accessible by anyone

  ✓ Correct Option B: Pre-Signed URLs (if access requires authentication)
    Generate time-limited URL server-side. User downloads via temp URL.
    S3 bucket: fully private. Access expires in N seconds.

PREVENTION:
  AWS Organizations SCP: deny s3:PutBucketPublicAccessBlock modification
  AWS Config rule: s3-bucket-public-read-prohibited → alert immediately
  S3 Account-level Block Public Access: enable (prevents any bucket from going public)
```

## SECTION 6 — System Design Importance

```
INCIDENT:
  SaaS company builds document management system.
  Versioning enabled (correct — protects against accidental deletion).
  No lifecycle policy for old versions.
  6 months later: S3 bill = $2,400/month. Expected: $200/month.

ROOT CAUSE:
  Application: users update documents frequently (avg 20 versions/document).
  Storage billed for: current + ALL previous versions

  Math:
    100K documents × 1MB average × 20 versions = 2,000GB of versions
    $0.023/GB × 2,000GB = $46/month per 6 months...
    But: documents grew, new users added
    Actual: 100TB of versions accumulated
    100TB × $0.023 = $2,340/month

FIX:
  Lifecycle rule: delete non-current versions after 90 days
  Lifecycle rule: delete delete markers (records of deleted versioned objects)

  # Terraform equivalent:
  lifecycle_rule {
    enabled = true
    noncurrent_version_expiration {
      noncurrent_days = 90
    }
    expiration {
      expired_object_delete_marker = true
    }
  }

ALSO ADD — Abort incomplete multipart uploads:
  Large file uploads that fail midway leave parts uploaded (billed, not visible as objects)

  expiration {
    abort_incomplete_multipart_upload {
      days_after_initiation = 7
    }
  }
```

## SECTION 7 — AWS & Cloud Mapping

```
INCIDENT (historical — before 2018 partition fix):
  Image processing pipeline: 500 workers upload to S3 simultaneously.
  Keys: "2024-01-15-001.jpg", "2024-01-15-002.jpg" ... (sequential timestamps)
  Performance: normal at 50 workers, degrades at 500 workers (latency 10× increase)

ROOT CAUSE (pre-2018):
  S3 older partition scheme: sequential key prefixes → same S3 partition.
  All 500 workers hitting same partition → throughput limit reached.

MODERN S3 (since 2018): auto-partitioning improved.
  S3 now handles 3,500 PUT/COPY/POST/DELETE + 5,500 GET/HEAD per second per prefix.
  Sequential keys no longer problematic for most workloads.

WHEN STILL RELEVANT (extreme scale):
  > 5,500 GET/sec on a single prefix:
  Old fix: hash prefix. Instead of "2024-01-15-001.jpg" → "a3f2-2024-01-15-001.jpg"
  Modern fix: use multiple prefixes explicitly (different "folders" per worker group)

ACTUAL MODERN S3 PERFORMANCE BOTTLENECK:
  Not key naming — it's Lambda concurrency + S3 trigger fan-out.
  High upload rate + S3 event notification + Lambda: Lambda concurrency limit (1000 default).
  Fix: use SQS between S3 event and Lambda (buffer + smooth concurrency spikes).
```

## SECTION 8 — Interview Preparation

**Beginner:**

**Q: What is S3 and what types of data should you store in it vs a database?**
**A:** S3 (Simple Storage Service) is AWS's object storage â€” you store files (objects) identified by a key (path-like string) in buckets. Unlike a database, there's no querying by content, no relationships, no transactions. S3 is: infinite scale, 11 nines (99.999999999%) durability, cheap storage (.023/GB), globally accessible. Use S3 for: user uploaded files (images, videos, documents), application logs, database backups, static website hosting, deployment artifacts (Docker images, Lambda packages). Use a database for: structured data you query, user accounts, orders, transactions â€” anything you need to filter/sort/join.

**Q: What is an S3 bucket policy and how is it different from IAM policies?**
**A:** IAM policies are attached to users/roles and define what AWS resources they can access. S3 bucket policies are attached to the BUCKET and define who can access that specific bucket. They work together: a request is allowed only if BOTH the IAM policy allows it AND the S3 bucket policy allows it (no explicit deny in either). Bucket policies are useful for: (1) Cross-account access (allow a different AWS account to read files). (2) Public access (allow everyone to read: "Principal": "*"). (3) IP-based restrictions. For internal apps where your ECS tasks access S3: use IAM roles on the tasks, no bucket policy needed. Bucket policies for: cross-account and public access only.

**Q: What does "S3 Block Public Access" setting do and should you enable it?**
**A:** Block Public Access is an account-level or bucket-level setting that prevents S3 objects from being accidentally made public â€” even if a bucket policy or ACL tries to grant public access. Enabled by default on new accounts. You should ALWAYS have this enabled at the account level for production. Only disable it for specific buckets that legitimately need public access (e.g., static website hosting, public CDN origin). A famous cause of data breaches: developer accidentally creates a public bucket policy, Block Public Access was disabled at account level, customer data exposed. Enable it at account level: ws s3control put-public-access-block --account-id {id} --public-access-block-configuration BlockPublicAcls=true,...

**Intermediate:**

**Q: What is S3 versioning and when should you enable it?**
**A:** Versioning keeps all past versions of every object. When you overwrite invoice-2024-01.pdf, the old version is preserved with a version ID. Enables: accidental deletion recovery (ws s3api delete-object --version-id xxx to permanently delete, or just "delete" creates a delete marker â€” restore by removing the marker), point-in-time recovery, audit trail. Enable versioning for: user-generated content you can't recover from elsewhere, critical application configurations, regulatory data. Downside: storage costs multiply (every version counts toward storage billing). Mitigate with: lifecycle rules to expire old versions after 30-90 days.

**Q: What are S3 storage classes and how do you use lifecycle policies to reduce costs?**
**A:** S3 storage classes by tier: *S3 Standard* (.023/GB) â€” hot data, frequent access. *S3 Standard-IA* (.0125/GB + retrieval fee) â€” infrequent access, accessed monthly. *S3 Glacier Instant Retrieval* (.004/GB) â€” archives accessed occasionally, milliseconds retrieval. *S3 Glacier Deep Archive* (.00099/GB) â€” long-term compliance archives, 12-hour retrieval. Lifecycle policy automates transitions: move to Standard-IA after 30 days â†’ Glacier after 90 days â†’ Deep Archive after 365 days â†’ expire after 7 years. For application logs: standard lifecycle can save 80% on cold log costs compared to keeping everything in Standard.

**Q: What is S3 multipart upload and when should your application use it?**
**A:** For files > 100MB, S3 recommends multipart upload: split the file into chunks (5MB-5GB each), upload parts in parallel, S3 combines them. Benefits: faster uploads (parallel), resume interrupted uploads (only retry failed parts), upload parts while the file is still being generated (streaming). In Node.js: AWS SDK's Upload construct handles multipart automatically for large files. Caveat: incomplete multipart uploads accumulate and cost money (parts uploaded but upload not completed). Set a lifecycle rule: "Delete incomplete multipart uploads older than 7 days" to prevent cost accumulation.

**Scenario 1:** Design the S3 architecture for a photo sharing platform where: (1) users upload photos (average 5MB), (2) photos are displayed to all users publicly, (3) photos must be stored for 5 years, (4) photos older than 6 months are rarely accessed, (5) the platform stores 1 million photos total.

- Bucket: photos-platform-prod with versioning disabled (originals are authoritative, no need to version).
