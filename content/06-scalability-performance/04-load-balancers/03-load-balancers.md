### Critical CloudWatch Metrics

```
ALB METRICS TO MONITOR:

RequestCount:
  What: Total requests processed.
  Use: Traffic baseline and growth.
  Alert: Spike > 200% of baseline → potential traffic event.

TargetResponseTime (p99):
  What: P99 latency from LB to target and back.
  Use: Latency SLO tracking.
  Alert: p99 > SLO threshold (e.g., > 500ms for 5 minutes).

HTTPCode_Target_5XX_Count:
  What: 5xx errors from backend targets.
  Use: Application error rate.
  Alert: > 1% of RequestCount sustained over 2 minutes.

HTTPCode_ELB_5XX_Count:
  What: 5xx errors generated by ALB itself.
  Meaning: ALB could not route to a healthy target.
  This number should be ZERO under normal operations.
  Alert: Any count > 0 for > 30 seconds.

HealthyHostCount:
  What: Number of targets passing health checks per target group.
  Alert: < minimum required (e.g., for 4-server group: alert if HealthyHostCount < 2).
  Critical: HealthyHostCount drops to 0 → full outage.

UnHealthyHostCount:
  Alert: Any value > 0 sustained for > 5 minutes.
  (Brief unhealthy during deploy → OK. Sustained → investigate.)

RejectedConnectionCount:
  What: Connections rejected because LB surge queue is full.
  Should be zero. High values: ALB capacity insufficient.
  Action: Verify ALB pre-warming plan is in place.

AWS X-RAY INTEGRATION:
  ALB → X-Ray traces (entire request path):
  Shows: ALB handoff time + Backend time + Response time broken down by segment.
  Identifies: whether latency is in ALB routing or in backend processing.
```

## SECTION 11 — Quick Revision

**Q: "Explain the difference between L4 and L7 load balancing. When would you use each?"**

> "Layer 4 load balancers operate at the transport layer — they see TCP and UDP headers (source IP, destination IP, ports) but not the payload contents. They route connections with extremely low overhead (~50 microseconds in NLB). Layer 7 load balancers inspect the full HTTP request — headers, path, cookies, body — enabling content-based routing, SSL termination, WAF rules, and sticky sessions.
>
> I use L7 (ALB) for HTTP/HTTPS APIs and web services — almost everything web-based. The additional routing intelligence is worth the small (~5ms) overhead. I use L4 (NLB) when I need non-HTTP protocols (TCP, UDP, MQTT), when clients need to whitelist a static IP, or when latency needs to be in the microsecond range — gaming servers, high-frequency trading, database TCP proxies."

---

**Q: "What is connection draining and why does it matter?"**

> "Connection draining — or in AWS, 'deregistration delay' — is the grace period given to a load balancer target being removed from rotation. When a server needs to come down (deployment, scale-in, manual removal), the load balancer stops sending NEW requests to it, but allows in-flight requests to complete for the duration of the drain window.
>
> Without it: a user's 10-second file upload request is cut off mid-stream when the server terminates. They get a connection reset. With proper drain: the server processes all in-flight requests to completion before shutting down.
>
> The setting matters in both directions. Too short (5s) and long requests are still cut off. Too long (300s, the AWS default) and scale-in events take 5 minutes per instance, making auto-scaling sluggish. Best practice: set drain timeout to 2× your P99 request duration based on actual access log data — typically 30–60 seconds for most web APIs."

## SECTION 12 — Architect Thinking Exercise

**Scenario:** Design the load balancing strategy for a multi-service platform:

- A REST API (HTTP, 500ms P99)
- A WebSocket service (persistent connections, real-time updates)
- A gRPC microservices layer (internal service-to-service, < 10ms P99)
- A legacy TCP service on port 5432 (non-HTTP, needs static IP for firewall rules)

---

**Answer:**

```
COMPONENT 1: REST API
  Load Balancer: ALB
  Listener: HTTPS :443
  TLS: Terminate at ALB (ACM certificate). HTTP to backend.
  Target Group: ECS Fargate tasks on port 8080
  Health Check: GET /health HTTP 200 (deep check — Redis + DB)
  Algorithm: Least Outstanding Requests
  WAF: Attached — rate limiting (1000 req/5min per IP), XSS/SQLi rules
  Routing: Path /api/* → this target group
  Drain: 60 seconds (2× P99 of 500ms → well above; standard 60s is fine)

COMPONENT 2: WebSocket Service
  Load Balancer: ALB (gRPC and WebSocket both use HTTP/2 — ALB handles both)
  Listener: WSS :443 (same ALB, different path: /ws/*)
  ALB idle timeout: EXTENDED to 3600s (WebSocket connections are long-lived;
                   default 60s would terminate active connections every minute)
  Target Group: ECS tasks

  INTERNAL STATE PROBLEM:
  WebSocket connections are stateful (pinned to one server).
  Solution: Redis Pub/Sub for message distribution.
    Any server receives a message for any user.
    Lookup: "which server is user X connected to?" → Redis
    Publish: message to channel "user:{userId}:inbound"
    Subscriber: whichever server holds that user's WebSocket, processes message.
  This allows elastic scaling of WebSocket servers without sticky sessions.

COMPONENT 3: gRPC Internal Services (east-west traffic)
  Load Balancer: NOT an external ALB.
  Use: AWS Cloud Map (service discovery) + ECS Service Connect

  Why not ALB for internal gRPC?
    ALB adds overhead for every internal call. At 10ms P99 target, the ~5ms
    ALB overhead is 50% of budget. Not acceptable.
    Internal services: use DNS-based service discovery or a service mesh sidecar.

  Option A (simpler): ECS Service Connect (built-in, no sidecar)
    Services register with Cloud Map. Others discover via DNS.
    Client-side load balancing via gRPC's built-in retry + load balancing.

  Option B (advanced): App Mesh (Envoy sidecar)
    Full service mesh: mTLS, retries, circuit breaking, canary weights.
    Overhead: higher ops complexity.

  Use Cloud Map + ECS Service Connect for gRPC if P99 < 20ms.
  Use App Mesh if circuit breaking + mTLS are required.

COMPONENT 4: Legacy TCP Service (port 5432, needs static IP)
  Load Balancer: NLB
  Reason: TCP protocol (not HTTP), static Elastic IPs needed for firewall allowlist.
  Listener: TCP :5432
  TLS passthrough: backend handles its own TLS (NLB passes raw TCP).
  Target Group: legacy EC2 instances
  Health Check: TCP (NLB sends TCP SYN, waits for ACK — no HTTP needed)

FULL ARCHITECTURE:

  Internet
    ├── ALB :443 → /api/*    → REST API target group (Fargate)
    │           → /ws/*     → WebSocket target group (Fargate)
    │
    └── NLB :5432            → Legacy TCP target group (EC2)
                               (Elastic IPs: 1.2.3.4, 2.3.4.5, 3.4.5.6  ← firewall rules point here)

  Internal:
    gRPC Services ←→ ECS Service Connect (Cloud Map DNS resolution)
```

## === Architect's Mental Model ===

### 5 Decision Rules

**Rule 1: Default to ALB for HTTP; default to NLB for everything else.**
This covers 95% of decisions. ALB has richer features for HTTP workloads. NLB is the right tool for non-HTTP protocols and for the rare cases where static IPs or microsecond latency are required. Going with CLB (Classic LB) is never the right answer for new infrastructure — it lacks path routing, SNI, and Lambda targets.

**Rule 2: Deep health checks, not trivial 200 responses.**
A health check that returns HTTP 200 unconditionally tells the load balancer nothing. The health check should verify the dependencies the server needs to actually process requests: can it reach Redis? Can it reach the database? If not, return 503. This turns your load balancer into an active resilience mechanism — it removes degraded instances from rotation automatically before they cause user-facing failures.

**Rule 3: Set deregistration delay based on your P99 request duration — not the default.**
The AWS default is 300 seconds. For most web APIs with P99 < 1 second, that's 300× too long. Measure: look at your ALB access logs for the 99th percentile of `target_processing_time`. Set drain to max(that × 2, 30s). Overly long drain makes your auto-scaling loop slow. Too short drain drops in-flight requests. Get this right during initial setup; it's rarely revisited and always impacts incident recovery time.

**Rule 4: Plan for LB capacity before traffic spikes, not during them.**
AWS ALB auto-scales but not instantaneously. If traffic spikes from 1,000 req/s to 20,000 req/s in 5 minutes (product launch, email blast), the ALB may not have scaled fast enough. Contact AWS support to pre-warm the ALB before known high-traffic events. Track `RejectedConnectionCount` metric: any non-zero value means clients are being refused before even reaching your backend.

**Rule 5: The load balancer is not the place to implement business logic.**
Path routing: fine at the LB layer. Authentication: also fine (OIDC at ALB). But business logic, complex auth policies, request transformation — belong in the application or an API Gateway layer, not in ALB listener rules. ALB listener rules are limited in expressiveness and are hard to test. Keep LB rules to routing, TLS, and basic security; keep business logic in code.

### 3 Common Mistakes

**Mistake 1: Treating the load balancer as the only layer of HA.**
"We have ALB, so we're highly available." ALB routes between healthy instances — it doesn't make your application logic HA. Database SPOF? ALB can't help. Redis single-node? ALB can't help. HA requires every component in the path to be HA: ALB (yes, AWS manages this), app servers (multiple instances in multiple AZs), database (Multi-AZ RDS), cache (ElastiCache with replication), DNS (Route 53 health checks with failover). The LB is one component of HA architecture, not the sum of it.

**Mistake 2: Not enabling cross-zone load balancing.**
Default behavior: ALB routes traffic only within the AZ the connection enters. If AZ-A receives 70% of traffic (due to client geography), AZ-A servers are overloaded while AZ-B servers are idle. Cross-zone load balancing lets ALB route to any healthy target across all AZs. For NLB, cross-zone is disabled by default and has a data transfer cost. For ALB, cross-zone is always on and free. Enable it. Verify in the target group settings that instance-level distribution is roughly equal across AZs.

**Mistake 3: Skipping access log analysis.**
ALB access logs contain the ground truth: who hit what URL, which backend served it, how long it took, what the response code was. Most teams enable access logs, write them to S3, and never look at them — until there's a production incident where they need to prove what happened. Set up Athena queries on ALB access logs from day one: "Top 10 slowest endpoints this week." "Which IPs are generating the most 5XX errors?" "What percentage of requests are failing health-check-induced 502s?" This analysis frequently surfaces performance problems before users file tickets.

### 30-Second Interview Answer

> "A load balancer is a single entry point that distributes incoming traffic across a pool of backend servers. It solves two problems: capacity (traffic is shared across multiple servers, any combination of which can be added or removed without changing the client-facing address) and availability (if any server fails its health check, the LB stops sending traffic there — users are served by the remaining healthy instances). At Layer 4 it routes TCP connections by IP/port with microsecond overhead; at Layer 7 it inspects HTTP headers enabling path-based routing, WAF, and sticky sessions with slightly higher overhead. In AWS: ALB for HTTP workloads, NLB for TCP/UDP or static IP requirements. The critical operational details: health checks must be meaningful (not just return 200), connection drain must be tuned to actual request duration, and cross-zone load balancing should be enabled for even distribution."
